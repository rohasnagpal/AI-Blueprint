AI Blueprint Feature Plan

Multi-Agent Contract Review Workflow

A comprehensive implementation plan for adding a new AI-assisted contract review workflow to AI Blueprint. This plan does not modify the existing contract reviewer persona. The feature is designed as a separate structured workflow with human lawyer oversight, auditable intermediate outputs, playbook comparison, clause-level review, and compatibility with local or private LLM deployments.

New feature No persona change Human oversight required Not autonomous legal advice Plugin-ready architecture

Executive Summary

The current contract review capability should be evolved from a single broad review call into a structured legal workflow. The new feature should run specialized modules in a controlled sequence, persist every important intermediate result, and give lawyers a clause-level review surface instead of a chatbot-style answer.

The system should keep the current contract reviewer persona untouched. Any existing persona remains available for chat, legacy review behavior, or comparison. The new workflow should live behind a new runner, new schemas, new storage, and new UI affordances.

Recommended implementation stance: Build an auditable workflow engine with LLM-assisted modules, not a fully autonomous multi-agent legal decision-maker. The system should help lawyers identify, compare, prioritize, and draft suggestions. It should not finalize legal advice or approve contracts.

Target outcome

  • Clause-by-clause extraction with source anchors.
  • Playbook comparison using organization-specific standards.
  • Risk scoring that prioritizes lawyer attention.
  • Redline suggestions that never auto-finalize edits.
  • Multiple audience summaries from the same reviewed evidence.
  • Escalation gates for critical and low-confidence issues.
  • Append-only audit trail for machine and human decisions.

Design Principles

Legal safety

AI-assisted, not autonomous

The workflow must consistently present outputs as analysis support. It must require human lawyer review for critical findings, missing protections, conflicts, and low-confidence outputs.

Traceability

Every conclusion needs a path

Each extracted clause, playbook comparison, risk level, redline suggestion, and summary should be traceable back to source text, agent version, prompt version, model, timestamp, and human decision history.

Modularity

Focused modules beat giant prompts

Each module should have a narrow task, bounded context, a strict input schema, a strict output schema, and an independently testable implementation.

Private LLM support

Provider-neutral execution

The workflow should use AI Blueprint's existing configured LLM abstraction so hosted, local, and private providers remain supported.

Deterministic where possible

Rules before reasoning

Playbook checks, required clause detection, prohibited clause matching, status transitions, and approval gates should use deterministic rules where possible. LLMs should handle semantic classification and drafting.

Extensibility

Custom agents later

The initial version can use built-in modules, but the design should prepare for configurable custom agents, custom playbooks, custom risk policies, and plugin-provided workflow stages.

Target Architecture

The feature should be implemented as a new workflow path for contract review blueprints. The existing contract reviewer persona should not be edited, renamed, deleted, or made responsible for this orchestration. If the legacy contract review runner remains available, the new workflow can be introduced as a new mode or new blueprint capability.

1. Intake Classify contract, parties, dates, governing law, category, routing.
2. Clause Extraction Segment, classify, anchor, and normalize contract clauses.
3. Playbook Comparison Compare clauses to approved, fallback, prohibited, and required standards.
4. Risk Scoring Assign low, medium, high, or critical risk with concise reasoning.
5. Redline Suggestions Suggest fallback wording for lawyer review only.
6. Summaries Create lawyer, business, and client-facing summaries.
7. Escalation Force human review for critical, conflicting, missing, or uncertain issues.
8. Human Review Approve, reject, comment, override, export, and preserve history.

Workflow Orchestrator Runs the stage graph, handles progress, cancellation, retries, and persistence. New runner module, for example contract_review_workflow_runner.py.
Agent Modules Perform narrow tasks with strict input and output contracts. New package, for example app/core/contract_agents/.
Playbook Engine Applies organization standards to extracted clauses. Hybrid deterministic rules plus optional semantic LLM comparison.

Audit Records machine outputs, confidence, timestamps, user decisions, and exports. Use existing audit events and skill runs, plus new review-history tables.

Implementation Prerequisites

These decisions should be locked before Phase 1 implementation. They affect the database migration, runner skeleton, extraction strategy, and review UI.

Required

Source anchoring upgrade

Clause-level review requires stronger source metadata than a simple chunk index. The document indexing path should preserve page number where available, chunk ID, chunk index, document ID, filename, character offsets within the extracted text, and enough excerpt text to verify the clause.

Required

Default playbooks

The first deployment should include at least two built-in playbooks: a general NDA playbook and a general MSA/vendor agreement playbook. Without these, playbook comparison will degrade to "unavailable" too often.

Required

Parallel execution choice

Use asyncio.gather inside the background job for concurrent module calls. This matches the existing council runner pattern and avoids introducing Celery or a new task queue in the first version.

Required

Long-contract extraction strategy

Long contracts should be processed in overlapping windows, then merged. The extractor should run over section-sized windows, deduplicate clauses by source span and semantic type, merge partial clauses, and keep source anchors for every merged result.

Decision Chosen approach Why
Workflow run storage Extend ContractReviewRun with nullable workflow fields. A separate 1:1 extension table adds joins without clear benefit for the first version.
Parallelism Use asyncio.gather for safe concurrent calls inside one background job. It fits the current architecture and keeps deployment simple.
Playbook baseline Seed default NDA and MSA/vendor playbooks. The comparison stage needs real standards on day one.
Source anchors Update indexing metadata before relying on page or offset UI. Clause review is only credible if findings are traceable to the contract text.

Agent and Module Plan

In this plan, "agent" means a bounded workflow module. Not every module should be a free-form LLM. Some should be deterministic services with optional LLM assistance. Each module should have a version, schema, confidence model, and clear failure behavior.

Module Inputs Outputs Execution style Failure behavior
Intake Agent Document text, document metadata, linked blueprint config. Contract type, category, parties, dates, governing law, jurisdiction hints, workflow route. LLM-assisted classification with deterministic metadata extraction where possible. Continue only if document is plausibly a contract. Otherwise escalate and stop.
Clause Extraction Agent Raw text, chunks, source locators, intake result. Normalized clause list with type, text, source anchor, confidence, and extraction notes. LLM-assisted segmentation plus rule-based post-processing. Stop or escalate if extraction quality is too low. Do not fabricate clause coverage.
Playbook Comparison Agent Extracted clauses, selected playbook, intake category. Per-clause approved, fallback, prohibited, missing, deviation, and standard language references. Deterministic required/prohibited checks first, semantic LLM comparison second. Continue with "playbook unavailable" status only if no matching playbook exists.
Risk Scoring Agent Clauses, playbook findings, contract category, organization risk policy. Risk level, likelihood, impact, concise reasoning, attention priority. Rules plus LLM explanation. Severity thresholds should be configurable. Unknown risk becomes medium or high with human review, never falsely low.
Redlining Suggestion Agent Flagged clauses, playbook standards, risk findings. Suggested replacement language, fallback language, negotiation rationale. LLM drafting constrained by playbook text and organization style rules. Omit suggestion if confidence is low or playbook support is missing.
Summarization Agent Intake, clause findings, risk scores, redline suggestions, escalation state. Lawyer summary, business summary, client-facing explanation, obligations, negotiation points. LLM summarization over validated intermediate data, not raw contract alone. Can degrade gracefully if redlines fail, but must disclose incomplete sections.
Escalation Agent All findings, confidence scores, conflicts, missing clauses, workflow errors. Escalation records with severity, reason, required action, linked clauses. Mostly deterministic gate with LLM support for ambiguity explanation. Always err toward human review when confidence, conflict, or coverage is uncertain.

Non-goals for the first version

  • No automatic Word redlining into the source document.

  • No hard dependency on OpenAI-only structured output features.

The existing coarse contract review output should remain for compatibility. New runs should additionally write clause-level and step-level records. Human review actions should be append-only so the review history can be reconstructed.

Document indexer metadata prerequisite

Before clause extraction can support reliable source anchors, indexed chunks should carry richer metadata. At minimum, the index should preserve document_id, chunk_id, chunk_index, filename, page when known, start_offset, end_offset, and extraction method metadata. If page numbers are unavailable for a file type, the UI should show chunk and excerpt locators instead of pretending page precision exists.

Built-in playbook seed content

Playbook Required baseline clauses Initial checks
General NDA Confidentiality, exclusions, permitted disclosures, term, return/destruction, remedies, governing law, assignment. Missing term, overbroad confidentiality, no exclusions, unilateral obligations, injunctive relief risk, missing governing law.
General MSA/vendor agreement Payment, scope, acceptance, warranties, limitation of liability, indemnity, confidentiality, IP, termination, data/security, dispute resolution, governing law, assignment, force majeure. Unlimited liability, uncapped indemnity, broad IP assignment, weak termination rights, payment ambiguity, missing data obligations, prohibited non-compete.

Persistence rule: Do not store only final prose. Store the underlying clause records, playbook findings, risk findings, and review decisions. The final memo should be reproducible from structured records plus summaries.

Orchestration Plan

The orchestrator should replace the all-in-one review flow for the new feature mode. It should operate as a background job, emit progress events, enforce cancellation, and write a step output after every module.

Load run, config, linked documents, and playbook

Resolve the blueprint, selected documents, selected playbook, model settings, organization risk policy, and review depth. If no indexed documents are linked, fail early with a clear message.

Build source bundle

Gather full text, chunks, document metadata, and source locators. Preserve enough locator data for clause review. This depends on the document indexer storing page, offset, chunk, filename, and extraction metadata wherever those values are available.

Run intake

Classify the contract and route to the appropriate review workflow. Select the best playbook based on contract category, governing law, workspace defaults, and user override.

Run clause extraction

Segment and classify clauses. For long contracts, process overlapping windows of roughly 8,000 characters with overlap, then deduplicate by source span, heading, clause type, and semantic similarity. Persist merged clauses immediately so later steps can link findings to stable clause IDs.

Run playbook comparison and risk scoring

Playbook comparison should run before final risk scoring. Certain deterministic risk rules can run in parallel, but the final risk score should account for playbook status.

Run redline suggestions for flagged clauses

Generate suggestions only for clauses that need attention. Suggestions must be stored separately from source text and marked as drafts requiring human approval.

Run summaries

Produce lawyer, business user, and client-facing summaries from validated structured outputs. Summaries should disclose unresolved escalations and incomplete sections.

Run escalation detector and finalize status

Create escalation records for critical risk, missing required clauses, conflicting provisions, low confidence, module failures, or ambiguous legal posture. Mark the run complete only as an AI draft review, not as lawyer-approved.

Sequential

Must run in order

Intake, clause extraction, playbook comparison, final risk scoring, escalation gating.

Parallel

Can run concurrently

Use asyncio.gather for clause-specific redlines, audience summaries, and deterministic risk prechecks after extraction.

Stop gates

Should not continue blindly

Invalid document type, failed extraction, missing source anchors, corrupted playbook, or unsafe output validation failures.

Long-contract extraction algorithm

  1. Split text into legal-structure windows using headings, numbering, and paragraph boundaries where available.
  2. Fallback to overlapping character windows when legal headings are unavailable.
  3. Extract clauses from each window with source locators and confidence scores.
  4. Deduplicate overlapping results by source span, clause type, heading, and normalized text similarity.
  5. Merge partial clauses and preserve all contributing source anchors.
  6. Flag low-confidence merged clauses for human review instead of treating them as complete.

API Plan

The current contract review run endpoints can remain. New endpoints should expose playbooks, clause-level findings, human decisions, and run summaries. All mutation endpoints should enforce blueprint permissions and record audit events.

Endpoint Purpose Notes
GET /contract-review/playbooks List playbooks available to the workspace or blueprint. Filter by category, jurisdiction, status, and version.
POST /contract-review/playbooks Create a workspace playbook. Owner/editor only. Record audit event.
PUT /contract-review/playbooks/{id} Update a playbook draft or create a new version. Prefer versioning over destructive edits to active playbooks.
POST /contract-review/runs Start a new workflow run. Add mode/config support for the multi-agent workflow.
GET /contract-review/runs/{run_id} Get run, final output, status, and summary. Remain backward compatible with existing output shape.
GET /contract-review/runs/{run_id}/clauses List extracted clauses and associated status. Support filters for risk, clause type, review status, and escalation.
GET /contract-review/runs/{run_id}/clauses/{clause_id} Get full clause detail. Include source anchor, playbook finding, risk finding, redline suggestion, and decisions.
POST /contract-review/runs/{run_id}/clauses/{clause_id}/decisions Approve, reject, request revision, or add note. Append-only. Never overwrite source clause or AI suggestion.
GET /contract-review/runs/{run_id}/trace Show module outputs and workflow events. Useful for debugging, audit, and quality review.
GET /contract-review/runs/{run_id}/export Export lawyer-ready report. Include unresolved escalations and human review status.

UI Plan

The UI should be run-centric. The lawyer should land on a selected contract review run and move through clauses, findings, suggestions, and decisions without relying on a chat transcript.

New screens should reuse AI Blueprint's existing UI language: the same navigation model, cards, tables, badges, buttons, modal behavior, typography, spacing, and restrained work-focused visual style. This feature should feel like a deeper Contract Review blueprint screen, not a separate product bolted onto the app.

Clause-by-clause review panel

  • Table or split list of extracted clauses.
  • Clause type, title, source, confidence, risk badge, playbook status.
  • Filters for high-risk, missing, prohibited, unresolved, and reviewed.
  • Selected clause detail with source text and document locator.

Risk heatmap

  • Clause types grouped by risk level and priority.
  • Counts for low, medium, high, and critical findings.
  • Clicking a cell filters the clause list.
  • Escalated issues are visually distinct from ordinary high risk.

AI vs playbook comparison

  • Extracted clause text on one side.
  • Playbook standard, fallback, and prohibited markers on the other.
  • Deviation summary and confidence score.
  • Redline suggestion shown as a draft only.

Human approval controls

  • Approve clause, reject suggestion, request revision, add note.
  • Decision history visible per clause.
  • Reviewer identity and timestamps recorded.
  • Critical escalations cannot be hidden by accidental clicks.

Escalation sidebar

  • Sticky list of critical and unresolved issues.
  • Links directly to affected clauses.
  • Required action shown plainly.
  • Resolve/dismiss controls based on permissions.

Summary tabs

  • Lawyer summary.
  • Business user summary.
  • Client-facing explanation.
  • Key obligations, negotiation points, unusual terms, payment and renewal risks.

Left rail

Audit and Traceability Plan

Audit should be treated as a product requirement, not an afterthought. The system should be able to answer: what did the AI see, what did each module produce, what did the human reviewer change or approve, and what was exported?

Step outputs should be written immediately after each module completes. They should not be batched at the end of the run, because partial results, failures, cancellation, and retries all need their own trace records.

Machine traceability

  • Workflow version and module versions.
  • Prompt template version per module.
  • Provider, model, temperature, token limits, and runtime metadata.
  • Input and output JSON for each module.
  • Confidence scores and validation errors.
  • Source documents, chunks, and excerpts used for each output.

Human traceability

  • Clause approvals, rejections, notes, and overrides.
  • Escalation resolution or dismissal decisions.
  • Reviewer user ID and timestamp.
  • Export events and exported report type.
  • Review status transitions.
  • Append-only decision history.
Important audit boundary: Do not rely only on a generic agent-output table as the legal audit trail. Use existing audit events for user actions, skill runs for module execution, and dedicated review-decision records for lawyer actions.

Rollout Plan

This should be delivered in phases to avoid destabilizing the existing app. The first version should preserve the current API response where possible while adding richer records and a new review UI.

Phase Scope Deliverables Acceptance criteria
0. Design lock Finalize schemas, failure policy, source-anchor metadata, async strategy, long-contract extraction, and playbook shape. Schema definitions, workflow contract, migration plan, default NDA/MSA playbook outline. All stages have typed inputs/outputs, clear stop/degrade behavior, and resolved implementation prerequisites.
1. Backend foundation Add data model, indexer metadata upgrade, migrations, module package, and workflow runner skeleton. Tables, schemas, richer source metadata, orchestrator, step recording. A run can execute with deterministic/mock modules and persist trace records with source anchors.
2. Extraction and playbooks Implement intake, windowed clause extraction, default NDA/MSA playbooks, and comparison. Extracted clauses, source anchors, seeded playbooks, playbook findings. Sample NDA and MSA contracts produce clause-level output with source references and playbook results.
3. Risk, redlines, summaries Implement risk scoring, suggestion drafting, and audience summaries. Risk findings, suggestions, summary records. High-risk clauses are prioritized and suggestions remain drafts.
4. Escalation and human review Add escalation gates and review decisions. Escalation records, approval/reject endpoints, decision history. Critical findings require visible human resolution.
5. UI implementation Build run-centric review screen. Clause panel, heatmap, comparison view, summaries, escalation sidebar. Lawyer can complete a review workflow without using chat.
6. Hardening Test long contracts, provider variability, validation failures, and exports. Smoke tests, fixtures, failure tests, export updates. Runs are reliable, auditable, and backward compatible.
7. Extensibility Add configurable custom agent registry and plugin hooks. Agent registry, stage config, plugin extension contract. New internal agents can be added without changing core orchestration code.

Risks and Mitigations

Risk Why it matters Mitigation
Latency increases Multi-stage review can require several LLM calls. Use background jobs, progress events, parallel clause-level work, and cached playbook comparisons.
False confidence Legal users may over-trust structured output. Show confidence, source anchors, unresolved escalations, and clear human-review status.
Weak source anchoring Findings are hard to verify without locators. Persist document, chunk, page, and offset data wherever available.
Playbook drift Organizations need versioned standards. Version playbooks and store selected playbook version on each run.
LLM output variability Local and hosted models may produce inconsistent JSON. Use schemas, validation, repair attempts, fallback statuses, and deterministic gates.
Over-agentizing Too many LLM calls can create complexity without value. Keep deterministic services deterministic; use LLMs only where semantic judgment or drafting is useful.
Accidental legal advice posture The product must not imply autonomous legal approval. Use product language, status flags, exports, and gates that require lawyer review before external use.

Build Order Recommendation

Start with backend structure and data persistence before the UI. The most important architectural shift is not the number of agents. It is making the workflow clause-level, schema-validated, auditable, and reviewable by a human lawyer.

  1. Lock schemas and table design.
  2. Lock source-anchor metadata and update the indexer plan.
  3. Choose asyncio.gather for first-version parallel execution.
  4. Define default NDA and MSA/vendor playbooks.
  5. Define the long-contract windowing, extraction, and merge strategy.
  6. Add workflow runner skeleton with mock modules.
  7. Add clause extraction with source anchors.
  8. Add playbook CRUD and seeded default playbooks.
  9. Add playbook comparison and risk scoring.
  10. Add redline suggestions and summaries.
  11. Add escalation gates and decision history.
  12. Add run-centric UI panels.
  13. Harden with tests, fixtures, exports, and provider compatibility checks.
  14. Add custom agent registry once the built-in workflow is stable.
