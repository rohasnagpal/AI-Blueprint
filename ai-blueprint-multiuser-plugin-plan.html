AI Blueprint Rebuild Plan

Server-mode, multi-user, plugin-powered blueprints

A comprehensive architecture plan for rebuilding AI Blueprint as a local/private server platform where admins enable capabilities, users create blueprints, and each blueprint combines personas, knowledge, credentials, workflows, and activity into one purpose-built AI workspace.

Decision

Rebuild the architecture, not the product idea

The current app is a strong prototype for workflows, but its core assumptions are now wrong for the target product: one user, one global workspace, one static HTML file, global settings, global documents, and every feature always present.

The new target is a server-mode product from day one: admin setup, multiple users, permissions, plugin enablement, and user-created blueprints.

Keep the visual identity, workflow learnings, prompts, built-in personas, model registry concept, RAG experiments, email flow, and council ideas. Do not keep the current schema or one-page frontend as the foundation.

Non-negotiables

  • Admin and users now, not later.
  • Blueprints are the primary user object.
  • Plugins define blueprint types and capabilities.
  • Personas and knowledge are reusable core modules.
  • UI should look familiar, but implementation must be modular.
  • All data access must be scoped by user, workspace, and blueprint permissions.

Core Modules vs Plugins

Do not make every concept a plugin. Some capabilities are platform primitives used by many plugins. Plugins should compose these primitives into useful blueprint types.

Module Type Reason Examples
Users / Auth Core Every route and object depends on identity. Login, sessions, password reset, invite users.
Workspaces Core Server installs need shared and private areas. Team workspace, private workspace, membership.
Permissions Core Controls plugins, blueprints, and dangerous actions. Admin, editor, runner, viewer.
Personas Core Reusable behavioral roles across plugins. Student Counsellor, Customer Support, Arbitrator.
Knowledge / RAG Core Many plugins need docs, links, retrieval, and citations. Uploads, URL ingestion, vector indexes, citations.
Models / Providers Core All AI workflows need provider/model selection. OpenAI, Anthropic, Groq, Ollama, local models.
Secrets Core Credentials need uniform encryption, scoping, and access rules. API keys, IMAP password, SMTP password, webhook secrets.
Email Automation Plugin A specific workflow built from secrets, personas, knowledge, models, and runs. Email blueprints for student counselling or support.
AI Council Plugin A specific multi-agent workflow builder. Arbitration, contract review, investment committee.

Blueprints

Definition

A blueprint is a configured AI workspace created from a plugin. It is not just a template. It is the place where the user configures the workflow, adds knowledge, selects personas, stores credentials if needed, runs the automation, and reviews outputs. 

Blueprint = {
  plugin type,
  name,
  workspace,
  members and permissions,
  personas,
  documents and links,
  credentials or secrets,
  workflow configuration,
  model settings,
  activity history,
  outputs and audit events
}

Blueprint permissions

  • blueprint.view
  • blueprint.edit
  • blueprint.run
  • blueprint.manage_members
  • blueprint.delete
  • Plugin-specific actions such as email.send or council.edit_flow.

Email Blueprint Example

Student Counsellor Inbox

  • Plugin: Email Automation.
  • Persona: Student Counsellor.
  • Credentials: user's IMAP/SMTP secret records.
  • Knowledge: admissions docs, scholarship docs, visa FAQ links.
  • Rules: draft only by default, escalate sensitive issues, never invent policy.
  • Activity: imported messages, generated drafts, sent replies, audit trail.

Customer Support Inbox

  • Plugin: Email Automation.
  • Persona: Customer Support Agent.
  • Knowledge: product docs, pricing, refund policy, known issues.
  • Rules: require approval before refunds or discounts.
  • Categories: billing, bug report, login issue, cancellation, feature request.
  • Activity: triage labels, drafts, send status, unresolved escalations.

Council Blueprint Example

Arbitration Blueprint

  • Plugin: AI Council.
  • Personas: Claimant Advocate, Respondent Advocate, Neutral Arbitrator, Award Drafter.
  • Knowledge: pleadings, contract, exhibits, procedural orders, links.
  • Flow: summary, argument, rebuttal, evidence weighing, draft award.
  • Output: issue list, evidence map, risk analysis, proposed award memo.

Contract Review Council

  • Personas: Commercial Lawyer, Risk Officer, Negotiator, Plain-English Summarizer.
  • Knowledge: contract, playbook, fallback clauses, prior agreements.
  • Flow: red flags, business impact, negotiation strategy, markup memo.
  • Output: clause table, recommended edits, questions for client.

Personas as Behavioral Roles

Personas should be reusable behavioral roles, not only chat styles. A persona can power email drafting, council agents, document chat responses, content generation, and support workflows.

Persona fields

  • Name and description.
  • Role identity and responsibilities.
  • Tone and response style.
  • Categories or routing labels.
  • Approval and escalation rules.
  • Knowledge-use rules.
  • Forbidden claims/actions.
  • Output format preferences.

Student Counsellor persona

Tone:
Warm, reassuring, clear, non-judgmental.

Categories:
Admissions, deadlines, visa concerns, course selection,
financial aid, housing, emotional distress.

Approval rules:
- Draft only; never auto-send unless explicitly allowed.
- Escalate self-harm, legal, medical, or immigration-risk messages.
- Human review before deadline, fee, scholarship, or policy claims.

Knowledge rules:
- Use uploaded university policy docs first.
- If policy is missing, say a staff member should verify.
- Never invent admissions requirements.

Plugin Roadmap

MVP should include a small number of plugins, but the platform should be designed so future plugins are natural. Core modules should remain reusable across plugin types.

Ownership and isolation rules

  • Every user-created object stores workspace_id.
  • Every user-created object stores created_by_user_id.
  • Blueprint data is always filtered by blueprint membership or workspace role.
  • Secrets are never returned in plaintext to the frontend.
  • Knowledge retrieval must filter by blueprint/workspace scope.
  • OpenAI vector stores should be per workspace or per blueprint, not global.
  • Local vector collections should be per workspace or per blueprint.
  • Audit events record sensitive actions: sends, deletes, plugin enables, permission changes.

Backend principles

  • Use explicit migrations from the beginning.
  • Protect every non-login API route.
  • Keep plugin manifests declarative.
  • Keep service logic separate from route handlers.
  • Use dependency-injected permission checks.
  • Centralize secret encryption/decryption.
  • Centralize audit logging for sensitive operations.
  • Add background job lifecycle state from day one.

Frontend Plan

Do not redesign the UI. Preserve the current visual language, but stop putting everything in one HTML file. The HTML should become a shell and the app should be modular.

What changes

  • index.html becomes an app shell.
  • CSS moves to standalone files.
  • Each view becomes a JS/TS module.
  • Navigation comes from the backend.
  • Blueprint pages render based on plugin type.
  • Login and admin views are added.

What stays the same

  • Sidebar look and interaction model.
  • Cards, forms, buttons, tables, and spacing style.
  • Chat layout and document upload feel.
  • Settings visual pattern.
  • Council and email workflow concepts.
  • Claude-like warm, restrained visual direction.

Admin Mode

Users

  • Create users.
  • Invite users.
  • Disable users.
  • Reset passwords.
  • Assign workspace roles.

Plugins

  • Enable or disable plugins.
  • Set plugin-wide limits.
  • Grant plugin access to users/roles.
  • Choose which plugins can create blueprints.

System

  • Configure model providers.
  • Manage global secrets.
  • View audit logs.
  • Run backups.
  • Export diagnostics.
Admins should be able to enable Email Automation globally without seeing a user's IMAP or SMTP password. Secret ownership and secret visibility must be separate concepts.

Security and Operational Requirements

Area Requirement Notes
Authentication Required for all app APIs. Use secure session cookies for browser users. Add API tokens later if needed.
Authorization Every query and action checks workspace, blueprint, and plugin permissions. Test cross-user denial explicitly.
Secrets Encrypted at rest and scoped to owner/workspace/plugin. Never return plaintext secrets to frontend.
RAG isolation Vector indexes must not mix unauthorized documents. Use per-blueprint or per-workspace collections/stores.
URL ingestion Block internal, private, loopback, link-local, multicast, and metadata IPs. Validate redirect targets too.
Email sending Permissioned, audited, rate-limited, and admin-controllable. Draft-only should be a safe default.
Audit Record plugin enablement, permission changes, sends, deletes, and exports. Useful for local server admins and debugging.
Backups Documented backup/restore for DB, uploads, vector stores, and secret key material. Cannot launch server mode without this.

MVP Acceptance Criteria

  • Admin can complete first-run setup.
  • Admin can create users and assign roles.
  • Admin can enable Email, Council, and Research plugins.
  • User only sees plugins and blueprints they can access.
  • User can create an Email Blueprint with credentials, persona, docs, links, and drafting rules.
  • User can import email, generate a draft, review it, and send only if permitted.
  • User can create a Council Blueprint with agents, personas, phases, docs, and links.
  • User can run a Council Blueprint and review evidence and outputs.
  • User can create a Research Blueprint and chat with scoped docs/links.
  • Cross-user data access is denied by automated tests.
  • Blueprint-level permissions are enforced by automated tests.
  • Secrets are encrypted and never returned as plaintext.
  • RAG retrieval cannot leak documents across blueprints/workspaces.
  • Admin can view audit events for sensitive actions.
  • Backup and restore steps are documented and tested manually.

Do Not Do Yet

Do not build a plugin marketplace

Start with first-party internal plugins. Third-party plugins require sandboxing, signed manifests, dependency isolation, and security review.

Do not redesign the UI

The user's requirement is explicit: keep the UI. Improve architecture and modularity without changing the visual product unnecessarily.

Do not skip data isolation tests

Multi-user server mode lives or dies on permission boundaries. Tests must prove isolation.

Do not make RAG global

Knowledge should be scoped to blueprint/workspace unless explicitly shared.

Do not expose plaintext secrets

Even admins should not casually view user-owned mail credentials or provider keys.

Do not start with every plugin

Email, Council, and Research are enough to prove the platform. Add more after the core model works.

Open Product Decisions

Question Recommended default Why
Are blueprints private by default? Yes. Least surprising and safest for email, legal, and documents.
Can admins see user blueprint contents? Only with explicit workspace/admin policy. Local server admin power is useful, but privacy expectations must be clear.
Can plugins be enabled per workspace? Yes, after global enablement. Admin may enable Email globally but restrict it to specific teams.
Can users bring personal API keys? Allow both admin-provided and user-provided provider credentials. Local deployments vary. Some admins centralize billing; some users bring keys.
Should email support auto-send? No for MVP. Draft and manual send only. Auto-send raises operational, reputational, and permission risks.
SQLite or Postgres? SQLite for early local server beta, Postgres-ready schema and migrations. SQLite is simpler, but avoid choices that block Postgres later.
