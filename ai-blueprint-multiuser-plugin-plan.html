AI Blueprint v2 Base Plan

Legal blueprint platform for law firms and lawyers

Final base plan for rebuilding AI Blueprint as a server-mode, multi-user legal AI platform where law firms configure reusable blueprints from personas, auditable skills, knowledge, rules, credentials, and council workflows.

First market: law firms Primary object: legal blueprint Formula: Persona + Skills + Knowledge Mode: admin-managed server
Decision Architecture Personas Skills Blueprints Councils Data Model Guardrails Roadmap

Decision

Build v2 for legal work first

AI Blueprint v2 should start with law firms, lawyers, and legal teams. The current prototype proves the value of documents, chat, councils, personas, email, and model selection, but the v2 product should be rebuilt around legal blueprints, auditable skills, multi-user permissions, and matter-specific knowledge.

The central product promise: a lawyer does not open a generic chatbot. They open a matter-specific blueprint that contains the right legal persona, the right skills, the right knowledge, the right rules, and the right review workflow.

Non-negotiables

  • Admin and multi-user support from day one.
  • Skills are first-class, auditable, versioned task units.
  • Personas are behavioral roles, not task definitions.
  • Knowledge is scoped by workspace, matter, and blueprint.
  • Escalation/human review is a designed output.
  • Citations and evidence mapping are mandatory for legal work product.
  • The UI can preserve the current feel, but the implementation must be modular.

Reference Standard

MIRA is the reference shape for legal coverage: a catalogue of legal skills organised into functional groups and composed into legal teams or workflows. AI Blueprint should not copy another product, but it should use the same architectural insight: legal AI should be a process of defined skills, not a single open-ended chat box.

Functional group target

The long-term legal skill map should cover the 24 functional areas identified by Lexlegis/MIRA: research, drafting, review, analysis, chronology and bundling, compliance, tax, litigation strategy, transactional support, due diligence, translation, document comparison, contract management, regulatory tracking, judicial preparation, corporate governance, competition, employment, real estate, startup advisory, litigation finance, investigations, not-for-profit, and legal education.

Skill type target

  • Output skills: defined artefact generation such as tables, memos, drafts, summaries, chronologies.
  • Hybrid skills: guided legal working sessions where the user supplies inputs iteratively.
  • Conversational skill: limited fallback mode, not the default legal architecture.
  • Meta-reasoning skills: citation verification, adversarial review, evidence mapping, confidence calibration.

Council run controls

  • Every phase stores inputs, outputs, evidence, model, persona, and skill versions.
  • Parallel phases are allowed only where role independence matters.
  • All legal conclusions run through citation verification and adversarial review where applicable.
  • Runs stream status over SSE and persist job events for audit.
  • Every council output is review-first, never automatically sent to a client.

Escalation and Human Review

Escalation is a first-class platform concept. A skill can produce an escalation event instead of, or in addition to, a normal output.

Missing knowledgeNo playbook, no jurisdiction setting, missing exhibits.Request missing source or mark limitation in output.

Engineering Guardrails

These rules are build requirements for v2. They are not suggestions. Codex should treat this section as the implementation contract before writing application code.

Database and persistence

Rule Implementation requirement
MigrationsUse Alembic from day one. Every schema change has upgrade and downgrade. No ad-hoc PRAGMA table_info migrations in application startup.
SQLite WALEnable PRAGMA journal_mode=WAL and PRAGMA synchronous=NORMAL on SQLite connections.
ConnectionsUse context managers or session-scoped dependencies. No scattered open/close pairs.
ConfigDeployment config such as DB path, port, secret key location comes from env/config file, not DB.
SettingsRuntime settings use an allowlisted settings table with declared key, type, scope, default, and write permission.
SecretsCredentials live in a dedicated encrypted secrets table with ownership and access columns. Never mix secrets into settings.

Authentication and authorization

Concern Rule
AuthAuthentication is middleware. Default route state is protected. Public routes such as login, setup, and health opt out explicitly.
AuthorizationAuthorization is an explicit route dependency checking workspace, blueprint, plugin, and action permissions.
SessionsUse HttpOnly cookies, SameSite=Strict, Secure in production. Store hashed session tokens server-side with expiry. Do not use localStorage for session tokens.
Current user/api/auth/me returns the current user and initializes frontend state.
BootstrapAdmin setup is one-time. After admin creation, setup closes permanently. No hardcoded backdoor credentials.
RolesUse workspace roles and blueprint roles separately. A workspace admin is not automatically a blueprint owner.

Plugin and blueprint architecture

Rule Requirement
Manifest vs moduleManifest is JSON-serializable data. Module is Python code containing routes, services, and skills. Never conflate them.
Dependency directionPlugins import from core. Core reads plugin manifests but must not depend on plugin internals at runtime.
Enablement chainPlugin use requires global enablement, workspace enablement, and user permission.
Config splitGeneric blueprint table holds cross-plugin metadata only. Plugin-specific fields go in plugin config tables.
Disable behaviorDisabling a plugin suspends active blueprints, preserves data, and sends cancellation signals to active jobs.

Skills architecture

Rule Requirement
Typed unitA skill has id, name, description, category, owner plugin/core, input schema, output schema, prompt template, required knowledge tags, and output validation.
ComposableSkills do not directly call other skills. Blueprint orchestration passes outputs between skills.
TestableSkills must be testable without an LLM by validating prompt construction and output parsing against mock model responses.
Legal specificityLegal skills use real legal workflow schemas: CUAD parameters, pleadings structures, evidence matrices, authority tables.

Isolation and multi-tenancy

  • Every user-created object includes indexed workspace_id and created_by_user_id.
  • Every query touching user data requires explicit workspace or blueprint scope.
  • Repository/query helpers should enforce scope instead of relying on route authors to remember filters.
  • RAG collections are scoped by blueprint or workspace. Never use a global collection.
  • Collection names follow bp_{blueprint_id} or ws_{workspace_id}.
  • Documents follow blueprint membership and explicit sharing rules.
  • Cross-user denial tests are required for blueprints, documents, runs, secrets, and skill outputs.

Secrets and security

Area Rule
Secrets tableUse secrets(id, workspace_id, owner_user_id, name, encrypted_value, scope, created_at). Scope is user, workspace, or admin.
PlaintextAPIs never return decrypted secret values, even to admins. Admins can see existence, revoke, or rotate.
Encryption keyKey comes from SECRET_KEY or a dedicated path outside the project directory. Document backup requirements.
URL ingestionResolve hostnames and block loopback, private, link-local, multicast, unspecified, and metadata IPs before fetch and after redirects.
UploadsValidate MIME type server-side. Store files by content hash, not original filename. Original filename is metadata only.

Background jobs and async

  • Long-running council and skill workflows use a DB-backed job state machine: pending, running, completed, failed, cancelled.
  • FastAPI BackgroundTasks alone is not a durable job queue.
  • Every job has timeout, heartbeat, cancellation, and failure state.
  • Use asyncio.get_running_loop(), not deprecated get_event_loop().
  • Blocking file, SQLite, embedding, and vector operations run in explicit executors.
  • Parallel agent outputs are serialized through a lock or written in one batch after gather completes.

API and frontend rules

Area Rule
Error schemaAll errors use {"error": "...", "code": "SNAKE_CASE_CODE", "details": {}}.
SSE schemaAll streams use {"type": "token|source|error|done|status|progress", "content": "...", "metadata": {}}.
PaginationList endpoints return {items, total, page, page_size}. No hardcoded LIMIT 100.
URLsResource ownership is in the URL, e.g. POST /workspaces/{workspace_id}/blueprints.
NavigationNavigation is server-driven from /api/me/navigation. No hardcoded permission-sensitive nav.
Frontend authRouter checks auth on every transition. Unauthorized resource access shows 403.
Plugin UIBlueprint views load dynamically by plugin type. Config forms are generated from plugin JSON schemas.

Audit, observability, and testing

  • Audit log is append-only: audit_events(id, workspace_id, user_id, action, resource_type, resource_id, metadata_json, created_at).
  • Audit plugin enable/disable, permission changes, secret create/rotate/delete, email send, blueprint delete, document delete, council run start, and all admin actions.
  • Use structured JSON logs with request ID, user ID, workspace ID, level, event, and duration.
  • Every HTTP response includes X-Request-Id.
  • Unit tests cover skills, encryption, and permission logic.
  • Integration tests use a real fresh SQLite test DB.
  • Isolation tests prove cross-user and cross-workspace denial.
  • Every permission boundary has a denial test that asserts 403.

Operations

  • Backup surface: SQLite DB, uploads directory, vector index directory, encryption key.
  • LAN deployment works with configurable CORS origins or same-origin deployment.
  • Dependencies are pinned with exact versions or managed by a lock file.
  • /api/health reports DB reachability, migration state, disk space, and encryption key loaded.

MVP Acceptance Criteria

  • Admin can create firm workspace, users, matters, and roles.
  • Admin can enable Legal Review, AI Council, and Legal Research plugins.
  • Admins can browse personas and skills separately.
  • Skills are versioned and every skill run is auditable.
  • User can create a Contract Review Workspace for a matter.
  • User can upload contract, playbook, precedent clauses, and client risk notes.
  • Contract Review produces structured extraction, risk matrix, negotiation memo, and client summary.
  • User can create Arbitration Preparation Council with scoped pleadings/exhibits.
  • Council runs stream status and persist phase outputs/evidence.
  • All legal outputs include citations or explicit unsupported flags.
  • Escalations are visible with reason, severity, and required reviewer.
  • Cross-user, cross-workspace, and cross-matter data leaks are blocked by tests.
  • No final client communication is sent without human approval.
  • Alembic migrations are the only schema-change mechanism.
  • Auth is middleware and authorization is dependency-based.
  • All settings are allowlisted and all secrets are stored in the secrets table.
  • All list endpoints are paginated.
  • Every sensitive action is in the append-only audit log.
  • Dependency versions are pinned or locked.
  • Backup and restore instructions cover DB, uploads, vector index, and encryption key.
